# 安全政策（Security Policy）

如果你发现了可能影响本项目用户或部署安全的问题，请不要在公开渠道（Issue/讨论区/社交媒体）直接披露细节。

---

## 报告安全漏洞

建议按以下方式进行披露（优先级从高到低）：

1. **私密渠道联系维护者**：优先通过仓库托管平台提供的私信/私有披露机制联系维护者 `flowerrealm`。
2. **安全公告机制（如可用）**：如果托管平台支持 *Private Vulnerability Reporting / Security Advisory*，请使用该机制提交。

为提高定位效率，请在报告中尽量包含：

- 影响范围（哪些接口/功能受影响、是否需要认证、影响面有多大）
- 复现步骤（最小可复现请求/配置）
- 预期/实际行为与风险评估（数据泄露/权限提升/SSRF 等）
- 相关日志片段（请先脱敏，**不要**包含真实 token/key/邮箱等敏感信息）

---

## 披露原则与响应

- 我们会尽力在收到报告后尽快确认问题、评估影响并给出修复计划。
- 在问题修复与缓解措施落地前，请避免公开披露细节。

---

## 安全边界（重要）

本项目的设计包含以下约束（详情以文档与代码为准）：

- 文档与示例中的 Key/Token 均应使用占位符（如 `rlm_...`、`sk-***`），禁止提交真实密钥。
- 上游凭据属于高敏感数据：请对数据库权限、备份、磁盘/云盘加密与日志脱敏保持高标准。
