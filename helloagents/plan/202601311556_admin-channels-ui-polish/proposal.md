# 方案提案：上游渠道页面 UI 优化（弱化“查询”概念 + 修复 modal 灰幕问题）

## 背景

用户反馈 `/admin/channels`（上游渠道管理）存在两类明显体验问题：

1. **“查询”概念不合理**：渠道管理页出现“查询统计”入口，用户感知为“渠道为什么要查询”，产生理解成本。
2. **点击弹窗按钮后页面被灰色笼罩**：打开 modal 时出现灰色遮罩，但弹窗可能被遮挡/层级异常，导致体验像是“页面被锁死”。

本轮以 SSR 模板为基线（SSOT），同时按用户反馈做“更符合直觉”的 UI 调整。

## 页面差异计划

页面：`/admin/channels`
  原来的内容：`internal/admin/templates/channels.html`
    - 顶部为简单页头（非卡片化），右侧仅“新建渠道”按钮
    - “开始日期/结束日期/查询统计”是 **用量统计区间** 的入口（GET），并非渠道列表查询
    - SSR base 模板在 `DOMContentLoaded` 时会把所有 `.modal` append 到 `body`（修复 backdrop/层级问题）
  现在的内容：`web/src/pages/admin/ChannelsPage.tsx`
    - 页头做成了卡片 + 图标，且出现“返回”按钮，整体信息密度与 SSR 不一致
    - 用量区间表单被包装成卡片 + 解释文本 + “查询统计”按钮，更像“必须先查才有数据”
    - `BootstrapModal` 在 React 树内渲染，未对齐 SSR 的“modal 挂到 body”策略，可能触发遮罩层级异常
  差异：
    - 将“查询统计”重定位为 **可选的统计区间设置**（弱化“查询”语义，降低误解）
    - modal 渲染方式对齐 SSR：将 modal 挂到 `document.body`（Portal），避免 backdrop 盖住弹窗
    - 统计区间控件对齐 SSR：放回页头下方（row g-2），始终展开；不展示时区文本（避免干扰）
    - 渠道详情区去掉不必要的“文字外框（code/badge）”，减少占用空间（保留必要信息）
